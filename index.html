SystemJS

TV v2 Retrospective

What's changed, gotchas and FAQs

Created by Marc Fallows / marc@blinkbox.com

Why?

  • Avoid complicated requireConfig, dynamic paths
  • Avoid app redirects and reduce server round trips
  • Ability to mix and match modules
    • "Universal dynamic module loader"
    • ES6, AMD, CommonJS, Global Scripts
  • Smarter bundling (System builder)
  • Dependency cache (DepCache) for faster loading
  • JSPM?

Client side detection

  • Happens in identify.js (formerly userAgentOverride.js)
  • Stored in a cookie, associated with an app version
  • Allows all UI Capability settings to work for apps which require client side detection (Chromecast)

* Remember *

  • When developing, you may need to clear the identity cookie if you previously emulated a TV

Server-side loader configuration

Order matters for merging

  1. App config
    1. Main config
    2. Client device config (highest priority to the most specific)
    3. Server device config (UI Capabilities)
    4. Mock config (in order of how they appear in the query string)
  2. Relative config (for a relative URL)
    • Config merged again, same order as app config.

* Remember *

  • Config can influence appSettings
  • URL used to define relativeUrl and view
    • Beware of not including a trailing slash

Web.config changes

New endpoints, locations to search for views and config.

RequireJS vs. SystemJS

  • SystemJS expects ES6 "System" global
    • Polyfilled by ES6 Module Loader
  • SystemJS wraps globals automagically
  • SystemJS lets us use CommonJS, ES6 modules on top of AMD

* Remember *

  • Need to list 'require' as a dependency
    • 
define(['require'], function(require){
    //...
});
  • Uses Traceur if it needs to transpile
    • Traceur isn't currently part of our project!

New alias needed

  • For SystemJS apps, we go to /v now, not /app
    • https://tv.blinkbox.com/v/latest/
    • https://tv.blinkbox.com/v/unstable/
    • https://tv.blinkbox.com/v/2.5.2/
  • Continues to support /app for legacy apps

* Remember *

  • Beware of getting redirected from /v to /app
    • On a legacy branch
    • The relative view is not found
  • URL will not redirect, uses the loader's baseURL (files are still served at /app/X/)
